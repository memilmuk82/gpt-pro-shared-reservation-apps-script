# PROJECT_INSTRUCTIONS.md

## 프로젝트 개요

### 프로젝트명

GPT Pro 공동 사용 지원 시스템

### 개발 목적

부서 공용 GPT Pro 계정을 학교 업무 환경에서 안전하게 예약·관리하기 위한 내부 업무용 웹앱을 개발한다.

이 프로젝트의 핵심 목적은 다음과 같다.

- 공용 GPT Pro 계정 사용 시간 예약
- 현재 사용 중인 사용자 확인
- 예약 충돌 확인
- 미등록 사용자 등록 요청 및 관리자 승인
- 사용자 권한 관리
- 설정 및 안내 문구 관리
- 공용 계정 사용 시 개인정보·평가보안·학생부 자료 입력 방지 안내

### 기술 스택

Frontend

- Vanilla JavaScript
- Tailwind CSS
- Google Apps Script HTMLService

Backend

- Google Apps Script

Database

- Google Sheets

Authentication

- Google Workspace 계정
- `senedu.kr` 도메인 기준

### 운영 환경

- Google Apps Script Web App
- Google Sheets 기반 데이터 저장
- 학교 관리자 Google 계정 소유 구조
- 학교 내부 업무용 게시판과 함께 운영
- GPT 계정 ID/PW는 시스템에 저장하지 않고 업무게시판에서 별도 관리

## 코드 작성 원칙

### Apps Script

- 기존 Phase 구조를 우선 유지한다.
- 새 기능은 가능하면 Phase 단위 파일로 분리한다.
- 기존 함수 재사용을 우선한다.
- 불필요한 신규 파일 생성을 지양한다.
- 단, 파일이 지나치게 커져 역할이 흐려질 경우 역할 기준으로 분리할 수 있다.
- Google Sheets 헤더는 기존 함수와 테스트에 영향을 주므로 임의 변경하지 않는다.
- 서버 함수는 항상 권한 검증을 포함한다.
- 관리자 함수는 `requirePhase2Admin_()` 계열 검증을 우회하지 않는다.
- 일반 사용자 함수는 현재 접속자 기준으로 동작해야 한다.
- 테스트 함수는 실제 운영 데이터가 아닌 `TEST_` prefix 데이터를 사용한다.
- 테스트 데이터는 테스트 종료 후 정리한다.
- Apps Script의 실행 계정, 소유 계정, 접속 계정 차이를 고려한다.

### 클라이언트 코드

현재 클라이언트 구조는 다음 역할을 유지한다.

- `Client_State.html`: 전역 상태와 라우트 정의
- `Client_Utils.html`: 서버 호출, 공통 유틸, escape, 날짜/시간, 설정값 helper, 팝업 메시지 생성
- `Client_Icons.html`: SVG 아이콘 반환
- `Client_Components.html`: 반복 UI 컴포넌트
- `Client_Views.html`: 화면 단위 렌더링
- `Client_Handlers.html`: 이벤트 핸들러와 서버 호출 처리
- `Client_App.html`: 앱 초기화, 라우팅, 초기 데이터 로딩

클라이언트 작업 시 다음을 지킨다.

- `Client_*` 구조 유지
- 상태 관리와 UI 렌더링 로직 분리
- 서버 호출 로직은 핸들러에 둔다.
- 반복 UI는 컴포넌트 함수로 분리한다.
- 화면 전체 렌더링은 `Client_Views.html`에 둔다.
- 유틸 함수와 팝업 메시지 생성 함수는 `Client_Utils.html`에 둔다.
- 중복 코드 최소화
- 기존 전역 함수 방식을 유지한다.
- 네임스페이스 구조 전환은 별도 리팩토링 Phase에서만 검토한다.
- HTML 문자열 렌더링 시 사용자 입력값은 반드시 escape 처리한다.
- `innerHTML` 사용 시 신뢰할 수 없는 값은 반드시 `escapeHtml()`을 거친다.

### Index include 규칙

현재 각 `Client_*.html` 파일 내부에는 `<script>...</script>` 태그가 포함되어 있다.

따라서 `Index.html`에서 Client include 전체를 다시 `<script>` 태그로 감싸지 않는다.

include 순서는 다음을 유지한다.

```html
<?!= include('Client_Utils'); ?>
<?!= include('Client_State'); ?>
<?!= include('Client_Icons'); ?>
<?!= include('Client_Components'); ?>
<?!= include('Client_Views'); ?>
<?!= include('Client_Handlers'); ?>
<?!= include('Client_App'); ?>
```

`Client_Utils.html`가 `Client_State.html`보다 먼저 와야 한다.

이유:

```javascript
selectedReservationDate: getLocalDateInputValue(new Date())
```

위 코드가 `Client_State.html`에서 즉시 실행되기 때문이다.

### 리팩토링

- 기능 안정성 우선
- 테스트 통과 후 리팩토링
- 리팩토링만을 위한 구조 변경 금지
- 리팩토링 전 변경 대상과 영향 범위 제시
- 리팩토링 후 회귀 테스트 수행

- 네임스페이스 리팩토링은 즉시 진행하지 않고 필요성이 명확할 때 별도 Phase로 검토

## 보안 원칙

### 입력값 처리

- 모든 사용자 입력은 서버에서 검증한다.
- 프론트 검증은 UX 보조이며 보안 검증이 아니다.
- 서버 검증을 우회할 수 있다고 가정하고 방어적으로 작성한다.
- `escapeHtml()` 사용 원칙 유지
- GuideItems, Settings 등 관리자 입력값도 화면 출력 시 escape 처리한다.
- HTML 문자열은 실행하지 않고 텍스트로 표시한다.
- `innerHTML` 사용 시 신뢰할 수 없는 값은 직접 삽입하지 않는다.
- XSS 방지 우선
- 민감정보 입력 방지 안내를 유지한다.

### 권한 처리

- 권한 검증 우회 금지
- 관리자 정책 우회 금지
- 관리자 전용 함수에는 반드시 관리자 검증을 둔다.
- 사용자 본인 데이터 조회와 관리자 전체 조회를 구분한다.
- 미등록 사용자는 예약 기능을 사용할 수 없다.
- 등록 요청은 접속한 Google 계정 기준으로 처리한다.
- 테스트용 우회값은 관리자 검증 후에만 허용한다.

### 계정 정보 보안

- GPT 계정 ID 저장 금지
- GPT 비밀번호 저장 금지
- 예약 시스템에 공용 계정 비밀번호를 넣지 않음
- GPT 접속 링크, ID, PW는 업무게시판에서 관리
- 예약 시스템에는 업무게시판 참고 안내만 표시
- 인증번호 담당자 정보는 시스템에서 표시 가능

## 정책 유지 사항

### 관리자 정책

- 최소 1명의 활성 `admin` 유지
- 마지막 활성 관리자 비활성화 금지
- 마지막 활성 관리자 권한 제거 금지
- `admin`과 `subadmin`의 역할을 구분한다.
- `admin`은 시스템 소유자급 관리자이다.
- `subadmin`은 업무 보조 관리자이다.
- 관리자 정책 변경은 반드시 승인 후 진행한다.

### 사용자 정책

- 사용자는 `Users` 시트에 등록되어야 시스템을 사용할 수 있다.
- 이메일은 사용자 식별자이며 수정하지 않는다.
- `senedu.kr` 도메인 계정만 등록한다.
- 사용자는 활성/비활성 상태를 가진다.
- 비활성 사용자는 예약 시스템을 사용할 수 없다.
- 기존 예약 및 사용 기록은 사용자 비활성화로 삭제하지 않는다.
- 미등록 사용자는 등록 요청을 할 수 있다.
- 등록 요청은 관리자 승인 또는 반려 대상이다.
- 등록 요청 시 휴대폰 번호는 수집하지 않는다.

### 등록 요청 정책

- 등록 요청 상태는 `대기`, `승인`, `반려`를 사용한다.
- 동일 이메일의 중복 대기 요청은 허용하지 않는다.
- 이미 등록된 사용자는 등록 요청할 수 없다.
- 외부 도메인 계정은 등록 요청할 수 없다.
- 승인 시 Users에 자동 등록한다.
- 반려 시 사유를 기록할 수 있다.
- 승인/반려는 관리자만 수행한다.

### 예약 정책

- 예약 무결성 유지
- 작업명은 필수
- 작업 설명은 선택
- 사용 전 확인 사항 동의 필수
- 시작 시간은 종료 시간보다 앞서야 한다.
- 사용 시간은 30분 이상이어야 한다.
- 충돌 예약은 감지한다.
- 충돌이 있을 경우 조율 확인 후 저장할 수 있다.
- 충돌 확인 없이 중복 예약 저장을 허용하지 않는다.
- 예약 등록 시 서버에서 최종 충돌 검증을 수행한다.
- 자동 충돌 조회는 현재 구조에서 기본값으로 사용하지 않는다.
- 예약 상태 전환 흐름을 유지한다.
- 완료 예약은 재시작할 수 없다.
- 예약 취소와 관리자 삭제는 구분한다.

### 설정 정책

- Settings는 허용된 key만 수정 가능하다.
- 허용되지 않은 setting_key는 저장하지 않는다.
- 필수 설정은 빈 값으로 저장할 수 없다.
- 숫자 설정은 범위를 검증한다.
- Settings 일괄 저장은 all-or-nothing 구조를 유지한다.
- 하나라도 잘못된 값이 있으면 아무 설정도 저장하지 않는다.
- 설정 변경은 SettingsLog에 기록한다.
- 계정 ID/PW는 Settings에 저장하지 않는다.

### GuideItems 정책

- 사용자 안내 화면에는 active=true 항목만 표시한다.
- 관리자 화면에서는 active=false 항목도 조회 가능하다.
- sort_order 기준으로 정렬한다.
- HTML은 실행하지 않고 문자열로 표시한다.
- GuideItems 생성/삭제는 별도 승인 전까지 구현하지 않는다.
- GuideItems 일괄 저장도 Settings와 동일하게 all-or-nothing 원칙을 적용한다.

## 테스트 원칙

- 수정 후 테스트 수행
- 테스트 결과 명시
- PASS/FAIL 구분
- 회귀 테스트 우선
- 백엔드 로직 변경 시 테스트 추가 또는 기존 테스트 보강
- 테스트 통과 전 다음 Phase로 진행하지 않음

- 테스트 종료 후 테스트 데이터 정리
- 사용자 관리, 예약, 설정, GuideItems는 기존 기능 회귀를 확인한다.

테스트 보고 시 다음을 명시한다.

- 실행한 테스트 함수
- 전체 테스트 수
- PASS 수
- FAIL 수
- 실패 항목이 있다면 코드, 메시지, 원인
- 수정 후 재실행 결과

현재 전체 테스트 기준은 PROJECT_STATUS.md를 따른다.

9. 테스트 방법 제시

11. 다음 단계 제안

### 새 채팅창에서 반드시 확인할 사항

- 현재 기준 테스트 PASS 수
- 현재 진행 Phase
- 수정 대상 파일
- Client 파일이 `<script>` 태그를 포함하는지 여부
- Index include 순서
- 현재 미구현 기능
- 승인된 변경인지 여부
- 기존 정책을 변경하는 작업인지 여부

### 작업 중 금지 사항

- 파일 전체를 임의로 재작성 금지
- 기존 기능 삭제 금지
- 정책 변경을 코드에 몰래 반영 금지
- 테스트 없이 다음 단계 진행 금지
- 계정 ID/PW 저장 기능 추가 금지
- 보안 검증을 프론트에만 의존 금지
- 사용자가 승인하지 않은 리팩토링 금지
- 여러 기능을 한 번에 섞어서 수정 금지

### 권장 작업 방식

- 파일 하나씩 점검
- 수정 전/수정 후 코드 블록 제시
- 적용 후 함수 존재 여부 확인
- 수동 테스트 항목 제시
- 전체 테스트 실행
- PASS 확인 후 다음 Phase 진행

## 출력 및 응답 원칙

- 수정 파일과 함수명을 먼저 말한다.
- 변경 이유를 짧게 설명한다.
- 수정 전 코드와 수정 후 코드를 구분한다.
- 검증 방법을 반드시 제시한다.
- 구현 완료와 제안을 구분한다.
- 미확인 사항은 추측하지 말고 질문한다.
- 불필요한 장황한 설명을 피한다.
- 정책 변경은 반드시 별도 승인 대상으로 표시한다.
- 사용자 요청이 코드 출력이면 적용 범위를 명확히 표시한다.
- 브랜치나 다른 대화창에서 이어받는 경우 현재 기준점을 먼저 확인한다.

## 장기 작업 방향

- Apps Script + Google Sheets 운영 버전을 먼저 안정화한다.
- Phase 10 이전까지는 네임스페이스 리팩토링을 진행하지 않는다.
- 커스텀 Modal은 Phase 10에서 진행한다.
- 포트폴리오용 Flask 버전은 운영 버전 안정화 후 별도 프로젝트로 진행한다.
- 포트폴리오 버전은 SQLite 우선, PostgreSQL은 Docker Compose 배포 옵션으로 검토한다.
- 현재 운영 앱에 불필요한 기술 스택을 도입하지 않는다.

## 추가 출력 지침

이 지침은 코드 제안 시 토큰 낭비와 불필요한 전체 코드 생성을 줄이기 위해 적용한다.

### 코드 변경 출력 방식

- 파일 전체 코드 출력 금지
- 변경사항이 있는 경우에만 변경 전 코드를 출력한다.
- 변경 전 코드는 실제로 바뀌는 최소 블록만 출력한다.
- 변경 후 코드는 변경사항이 있는 최소 블록만 출력한다.
- 추가만 있는 경우에는 추가 위치를 먼저 설명한다.
- 추가만 있는 경우에는 변경 전 코드를 출력하지 않는다.
- 추가만 있는 경우에는 추가 코드만 출력한다.
- 사용자가 수동 수정할 수 있을 정도의 코드만 출력한다.
- 불필요한 새 파일 생성, 전체 diff 생성, 전체 파일 재작성은 피한다.

### 변경 안내 기본 형식

변경이 있는 경우:

```text
변경 파일
변경 함수/위치
변경 이유
변경 전 코드
변경 후 코드
검증 방법
```

추가만 있는 경우:

```text
변경 파일
추가 위치
추가 이유
추가 코드
검증 방법
```

### 문서와 실제 코드가 충돌할 때

- 실제 코드를 우선 확인한다.
- 문서가 오래된 경우 문서 기준으로 추측 구현하지 않는다.
- 새 Phase 시작 전 실제 Apps Script 최신 파일을 다시 확인한다.
- 코드 수정 전 변경 대상과 영향 범위를 먼저 제시한다.
